## 한 줄 요약
이 영상의 핵심은 **AI에게 일을 잘 시키는 문제를 프롬프트 문장력의 문제로만 보지 말고, 에이전트가 어떤 규칙을 읽고, 무엇을 기억하고, 어떤 절차를 재사용하고, 어디까지 강제될지를 설계하는 구조의 문제로 봐야 한다**는 데 있다.

## 왜 이 영상을 남겨두나
요즘 AI 코딩 도구를 쓰다 보면 금방 부딪히는 한계가 있다.

- 한 번은 잘 되는데 반복하면 흔들린다
- 프로젝트가 길어질수록 맥락이 흐려진다
- 매번 같은 설명을 다시 해야 한다
- 외부 도구 연결, 검증, 승인 같은 단계가 자연스럽게 붙지 않는다

이 영상은 바로 그 지점에서 **하네스 엔지니어링**이라는 말을 꺼낸다. 단순히 좋은 모델을 고르는 것보다, **에이전트가 어떤 방식으로 일하게 만들 것인가**를 구조로 설계하는 쪽이 더 중요해졌다는 메시지다.

## 공부한 것

### 1. 하네스 엔지니어링은 “AI가 일하는 방식을 구조로 고정하는 일”에 가깝다
영상은 하네스를 아주 실무적인 말로 설명한다. 프롬프트만으로는 앱을 만들거나 긴 작업을 안정적으로 반복시키기 어렵기 때문에, **에이전트가 원하는 방식대로 동작하도록 붙여 주는 구조물**이 필요하다는 것이다.

이 정의가 좋았던 이유는, 하네스를 거창한 새 기술보다 **반복 가능한 작업 방식을 만드는 틀**로 이해하게 해 주기 때문이다. 결국 핵심은 이런 쪽이다.

- 같은 기준을 계속 따르게 하고
- 자주 쓰는 절차를 재사용하게 하고
- 외부 도구를 연결하고
- 꼭 지켜야 하는 가드레일은 강제하고
- 필요하면 역할을 분리해 병렬로 일하게 만드는 것

즉 하네스 엔지니어링은 “AI에게 더 똑똑한 답변을 받는 법”보다, **AI가 덜 흔들리게 일하도록 작업 환경을 설계하는 법**에 더 가깝다.

### 2. 영상은 하네스의 구성 요소를 6가지로 정리한다
이 영상에서 가장 실용적인 부분은 구성 요소를 비교적 선명하게 나눈 점이었다.

#### 2-1. `AGENTS.md` / 규칙 파일
가장 먼저 나오는 것은 규칙 파일이다. 에이전트가 항상 참고해야 할 지시를 파일로 둔다.

영상의 포인트는 이 규칙 파일이:

- 전역으로도 둘 수 있고
- 프로젝트별로도 둘 수 있고
- 중첩 구조로도 운영할 수 있으며
- 팀과 함께 버전 관리해야 하는 자산

이라는 점이다.

이건 결국 “매번 채팅창에서 말하지 말고, 프로젝트가 알아야 할 기본 원칙은 코드베이스 안으로 끌어와라”는 뜻으로 읽힌다.

#### 2-2. 메모리
메모리는 세션을 넘어 이어지는 맥락이다.

- 사용자의 선호
- 반복 업무 흐름
- 기술 스택
- 자주 빠지는 실수
- 이후 세션에도 유용한 결정

같은 것들을 축적해 두고 다음 작업에서 다시 불러오게 하는 구조다.

여기서 중요한 건 메모리가 단순 로그가 아니라, **다음 세션에서 재사용할 수 있는 맥락의 압축본**이라는 점이다.

#### 2-3. 스킬
영상은 스킬을 프롬프트 템플릿보다 넓은 것으로 설명한다. 단순히 문장만 저장하는 게 아니라:

- 지시문
- 참고 자료
- 템플릿
- 스크립트

까지 묶어서 특정 작업을 안정적으로 반복하게 만드는 재사용 워크플로우라는 것이다.

이 설명이 특히 좋았던 이유는, 스킬을 “좋은 프롬프트 모음”이 아니라 **작업 절차의 모듈화**로 보게 만들기 때문이다.

#### 2-4. MCP
MCP는 외부 시스템과 연결되는 층이다.

영상에서는 Slack, Gmail 같은 예시를 들며, 에이전트 혼자서는 할 수 없는 일을 외부 도구 연결을 통해 가능하게 만든다고 설명한다. 결국 이 층이 있어야 AI가 채팅창 밖으로 나갈 수 있다.

즉 MCP는 단순 부가 기능이 아니라, **AI를 실제 도구 환경과 이어 주는 인터페이스 층**으로 보는 편이 맞다.

#### 2-5. 훅
훅은 영상에서 꽤 중요한 위치를 차지한다. 규칙 파일, 메모리, 스킬은 에이전트가 “읽고 따르길 기대하는 것”에 가깝지만, 훅은 다르다.

훅은:

- 세션 시작
- 프롬프트 제출
- 도구 사용 전
- 턴 종료

같은 런타임 이벤트에 맞춰, **프로그래밍 로직으로 강제**하는 층이다.

즉 “이건 꼭 지켜야 한다”를 자연어 설득이 아니라 시스템 레벨에서 밀어 넣는 장치다.

#### 2-6. 서브 에이전트
서브 에이전트는 별도 컨텍스트를 가진 독립 작업자다.

- 병렬로 속도를 올리거나
- 검증 전용 역할을 분리하거나
- 서로 다른 문제를 각기 다른 맥락에서 처리하거나

할 때 유용하다.

여기서 핵심은 메인 세션 하나에 모든 걸 몰아넣지 않고, **전문 작업을 분리해 다루는 구조**를 만들 수 있다는 점이다.

### 3. 이 6가지는 각각 따로가 아니라 “기준-절차-도구-강제”의 층위로 읽는 게 더 좋다
영상을 다 보고 나면, 이 요소들을 개별 기능으로 외우는 것보다 다음처럼 묶어 이해하는 편이 더 남는다.

- **기준을 잡는 층**: `AGENTS.md`, 메모리
- **절차를 묶는 층**: 스킬, MCP
- **실행을 통제하는 층**: 훅, 서브 에이전트

이렇게 보면 하네스 엔지니어링은 결국 “AI가 무엇을 참고하고, 어떤 순서로 일하고, 어디까지 자유롭고, 어디부터 강제되는가”를 점진적으로 설계하는 일이라는 게 더 분명해진다.

### 4. 영상이 강조한 실제 작업 흐름은 ‘인터뷰 → 플랜 → 구현 → 검토’다
이론 설명만 한 것이 아니라, 실습 흐름도 꽤 선명했다. 영상에서 보여 준 기본 패턴은 대략 이렇다.

1. **인터뷰 단계**  
   처음부터 완벽한 요구사항을 사람이 직접 쓰려고 하지 않고, AI가 질문하게 해서 요구를 구체화한다.

2. **플랜 작성**  
   인터뷰 결과를 바탕으로 계획 문서를 만든다.

3. **프로젝트 구조와 규칙 설정**  
   구현 전에 기본 구조와 규칙을 잡아 준다.

4. **구현 진행**  
   실제 코드 작업을 수행한다.

5. **검토와 보완**  
   필요하면 HTML 시각화나 브라우저 검증을 활용해 사람이 더 잘 리뷰한다.

여기서 특히 남는 포인트는, 영상이 **요구사항조차 AI와 대화하며 끌어낸다**는 점이다. 내가 뭘 원하는지 완전히 정리되지 않았을 때, 오히려 AI가 질문하게 해서 설계를 더 선명하게 만들 수 있다는 관점이다.

### 5. 플랜 문서를 HTML로 검토하자는 제안은 생각보다 중요해 보였다
영상 중간에서 꽤 흥미로웠던 부분은 이 지점이다. 계획 문서가 마크다운으로 길어지면, AI는 잘 읽어도 사람이 빠르게 검토하기는 어려울 수 있다는 것이다.

그래서:

- 긴 플랜 문서를
- 사람이 5분 안에 검토할 수 있게
- 단일 HTML 리뷰 화면으로 바꿔 보자

는 제안을 한다.

이건 단순 시각화 팁이 아니라, **AI가 잘 읽는 형식과 사람이 잘 검토하는 형식은 다를 수 있다**는 점을 인정하는 설계처럼 느껴졌다. 결국 하네스 엔지니어링은 에이전트만 위한 구조가 아니라, **사람이 중간에 판단하고 승인하는 루프까지 포함하는 구조**여야 한다는 뜻이다.

### 6. 실습 예제보다 더 중요한 건 ‘처음부터 완벽하게 만들려 하지 말라’는 메시지다
영상은 하네스를 처음부터 다 깔끔하게 설계하려고 하면 너무 어렵다고 말한다. 대신:

- 먼저 규칙을 잡고
- 반복 절차를 스킬로 묶고
- 필요할 때 외부 도구를 붙이고
- 꼭 필요한 곳에 훅을 넣고
- 전문 작업만 서브 에이전트로 분리하는 식으로

점진적으로 쌓아 가는 방식을 추천한다.

이게 현실적인 이유는, 실제 프로젝트에서는 처음부터 모든 반복 패턴이 보이지 않기 때문이다. 일단 부딪히고, 반복되는 문제를 발견하고, 그다음 구조로 끌어올리는 편이 훨씬 자연스럽다.

### 7. 실습 데모는 결국 “하네스가 앱을 만드는 속도”보다 “앱을 흔들리지 않게 만드는 방식”을 보여 준다
영상에서는 `Commit Hero`라는 예시 서비스를 만들며:

- 인터뷰로 요구 구체화
- 플랜 문서 작성
- 프론트엔드 구조 세팅
- 브라우저 검증
- GitHub 반영
- Vercel 배포
- 이후 워크트리/새 세션으로 AI 기능 확장

같은 흐름을 보여 준다.

여기서 핵심은 예제 앱 자체보다, **한 프로젝트를 여러 세션과 도구, 문서, 배포 단계로 이어 붙이는 흐름**이었다. 결국 하네스 엔지니어링은 한 번의 답변 품질이 아니라, **긴 작업을 계속 굴릴 수 있는 작업 시스템 설계**에 가깝다는 점이 다시 확인된다.

## 내가 가져갈 포인트
이 영상을 보고 남긴 핵심은 세 가지다.

1. **프롬프트만 잘 쓰는 것으로는 장기 작업이 안정되지 않는다.**  
   규칙, 기억, 절차, 도구 연결, 강제 로직이 함께 있어야 한다.

2. **하네스는 기능 목록이 아니라 작업 운영체제에 가깝다.**  
   `AGENTS.md`, 메모리, 스킬, MCP, 훅, 서브 에이전트는 각각 따로가 아니라 하나의 운영 구조를 이룬다.

3. **사람의 검토 루프까지 설계해야 진짜 실무형이 된다.**  
   인터뷰, 플랜, HTML 리뷰, 브라우저 검증 같은 단계는 AI만 잘 쓰는 법이 아니라 사람과 AI가 같이 일하는 법에 더 가깝다.

## 아직 헷갈리는 것
- 어떤 수준부터 훅으로 강제하고, 어디까지는 규칙 파일로 두는 것이 적절한지 더 사례가 필요하다.
- 메모리와 스킬의 경계를 실제 프로젝트에서 어떻게 나누는 것이 가장 깔끔한지도 더 실험이 필요하다.
- 서브 에이전트를 어디서부터 병렬화 이득이 나는 구조로 설계할지 역시 경험이 더 필요해 보인다.

## 나중에 더 볼 것
- 실제 프로젝트 하나를 골라 `AGENTS.md → 스킬 → 훅` 순서로 하네스를 단계별로 붙여 보기
- 긴 플랜 문서를 HTML 검토 화면으로 바꿔 보는 실험 해 보기
- 반복적으로 흔들리는 작업을 찾아 스킬과 훅 중 무엇으로 해결하는 게 맞는지 비교해 보기

## 작업 메모
이 영상은 “Codex 사용법” 영상이기도 하지만, 더 크게 보면 **AI 시대의 개발 환경을 어떻게 구조화할 것인가**에 대한 영상이었다.

특히 인상적이었던 건, 하네스를 처음부터 완성형 설계도로 보지 않고 **반복되는 문제를 발견할 때마다 규칙·스킬·도구·훅으로 끌어올리는 점진적 구조화 작업**으로 본 점이다.

그래서 이 메모를 남겨둔다. 앞으로 AI 개발에서 중요한 것은 “어떤 모델을 쓸까” 못지않게, **내가 반복할 일하는 방식을 어디까지 구조로 고정해 둘 수 있는가**일 가능성이 크기 때문이다.
