## 한 줄 요약
이번 글의 핵심은 **어느 모델이 더 똑똑한가**보다, 이제는 **AI를 어떻게 연결하고, 저장하고, 운영하고, 검수해서 실제 결과물로 바꾸는가**가 더 중요해졌다는 점입니다.

## 왜 이 글을 남겨두나
이번 주 제가 속한 커뮤니티에서 같이 볼 글이었는데, 단순히 링크를 많이 모아둔 큐레이션이라기보다 **지금 AI 흐름을 어떤 프레임으로 읽어야 하는지**를 잘 보여주는 글이었습니다.

특히 좋았던 점은 60개의 자료를 하나하나 외우게 만들지 않고,
그 자료들 안에서 반복되는 방향 전환을 먼저 보여준다는 점이었습니다.

예전에는 주로 이런 질문을 했습니다.

- 어느 모델이 더 잘 쓰나
- 누가 더 똑똑한 답을 하나
- 어떤 모델이 코딩을 더 잘하나

그런데 이 글은 그 질문만으로는 이제 부족하다고 말합니다.

지금 더 중요한 건,
- 목표 기반으로 움직이는가
- 외부 도구와 연결되는가
- 기억과 문서를 남기는가
- 검수 가능한 결과물을 만드는가
- 내 환경 안에서 계속 굴릴 수 있는가

같은 질문입니다.

그래서 이 글은 단순한 최신 동향 정리라기보다,
**앞으로 무엇을 공부해야 하는지 방향을 잡아주는 글**로 읽혔습니다.

## 공부한 것

### 1. 이제 AI 논의의 중심은 모델 비교만으로는 부족하다
이 글에서 가장 먼저 크게 남은 문장은, 한때는 “어느 모델이 더 잘 쓰나”가 중심이었지만 지금은 그 질문만으로 부족하다는 지점이었습니다.

정말 그렇다고 느껴졌습니다.

요즘은 좋은 모델이 너무 빨리 평준화되고 있고,
모델 자체의 똑똑함만으로 실제 업무 성과가 갈리지 않는 경우가 점점 많아지고 있습니다.

오히려 차이를 만드는 것은:
- 목표를 어떻게 주는가
- 반복 작업을 어떻게 묶는가
- 어떤 도구와 연결되는가
- 자료를 어떻게 기억하게 하는가
- 결과물을 어떤 포맷으로 마감하게 하는가

같은 **작업 시스템의 설계** 쪽입니다.

그래서 이 글은 새 모델 발표를 따라가는 글이라기보다,
**좋은 모델을 넘어서 좋은 AI 작업 환경을 만드는 문제**로 시선을 옮기게 합니다.

### 2. 코딩 AI는 답변기가 아니라 작업 운영 도구로 이동하고 있다
글에서 첫 번째 큰 흐름으로 다룬 부분이 바로 이것입니다.

예전의 코딩 AI는:
- 코드 조각을 추천하고
- 질문에 대한 답을 주고
- 에러 원인을 설명해 주는 도구에 가까웠습니다.

하지만 지금은 다릅니다.

이 글이 짚는 변화는,
코딩 AI가 점점 **목표를 받으면 그 목표를 향해 여러 단계를 계속 수행하는 루프형 시스템**으로 가고 있다는 점입니다.

예로 나온 키워드들도 그 흐름과 잘 맞습니다.

- Codex `/goal`
- AGENTS.md
- Crack CLI
- Chrome DevTools MCP for Agents

즉 AI는 한 번 답하고 끝나는 존재가 아니라,
**작업 맥락을 유지하면서 다음 단계를 이어가는 실행 주체**에 가까워지고 있습니다.

이건 단순히 성능 향상이라기보다, 사용 방식 자체가 달라지는 변화라고 느껴졌습니다.

### 3. Hermes 같은 흐름은 ‘좋은 서비스’보다 ‘내가 통제하는 런타임’에 가깝다
이 글에서 개인적으로 반가웠던 부분은 Hermes 계열을 하나의 큰 흐름으로 묶어서 본 대목이었습니다.

핵심은 단순합니다.

이제 사람들은 “어떤 AI 서비스를 쓸까?”보다,
**내가 계속 켜두고, 연결하고, 통제하고, 유지할 수 있는 작업 환경을 어떻게 만들까?**를 더 묻기 시작했다는 것입니다.

예시로 나온 것들도 이 방향을 잘 보여줍니다.

- Hermes Desktop
- Discord/Telegram 연결
- Oh My Hermes
- Discord Agent Bridge

이런 흐름은 AI를 소비하는 방식에서,
**AI를 직접 운영하는 방식**으로 넘어가는 느낌을 줍니다.

즉 중요한 건 모델 하나를 잘 고르는 것이 아니라,
내 환경 안에서 AI가 계속 일할 수 있도록 **런타임을 설계하는 능력**이 됩니다.

### 4. MCP와 브라우저 연결은 AI를 외부 세계에 붙이는 표준층처럼 보인다
이 글에서 또 하나 중요하게 읽힌 것은 MCP와 브라우저 연결 흐름이었습니다.

이제 AI가 대화만 잘해서는 부족합니다.
정말 일을 하려면:
- 웹을 보고
- 버튼을 누르고
- 양식을 채우고
- 외부 도구를 호출하고
- 브라우저 상태를 읽고
- 디버깅까지 해야 합니다.

글에서 예시로 든:
- PlayMCP
- Sigma Private AI Browser
- Chrome DevTools MCP for Agents

같은 사례는 바로 그 전환을 상징합니다.

즉 AI는 점점 **대화창 안의 답변기**에서,
**외부 세계와 연결된 실행 주체**가 되려고 합니다.

이 흐름을 보고 나니 MCP를 단순 프로토콜로만 볼 게 아니라,
AI가 현실의 도구와 접속하는 **표준 인터페이스 층**으로 이해해야겠다는 생각이 들었습니다.

### 5. 개인 지식 시스템의 핵심도 저장이 아니라 재사용으로 바뀌고 있다
글에서 네 번째 큰 흐름으로 묶은 지점도 인상적이었습니다.

예전에는 주로 “노트를 얼마나 잘 정리할까”가 중심이었다면,
이제는 **그 노트와 자료를 다시 AI 작업에 어떻게 투입할 수 있을까**가 더 중요해졌다는 이야기입니다.

묶여 나온 예시는:
- NotebookLM
- Obsidian
- Code Wiki
- autoresearch
- Karpathy의 Obsidian RAG

이 흐름의 포인트는 분명합니다.

많이 저장하는 사람보다,
**저장한 자료를 다시 일하게 만드는 사람**이 더 강해질 가능성이 커집니다.

이건 저한테도 꽤 직접적으로 들어오는 얘기였습니다.
블로그든 메모든 결국 중요한 건 쌓는 행위 자체보다,
그 기록이 나중에 검색되고, 연결되고, 다시 작업에 들어갈 수 있게 만드는 구조라는 점이기 때문입니다.

### 6. 공공 문서와 형식 엄격한 보고서 자동화가 오히려 실무형 AI의 본무대일 수 있다
화려한 이미지 생성이나 자유형 글쓰기보다,
이 글은 오히려 **형식이 빡빡한 문서 현장**이 생성형 AI의 진짜 시험장이 될 수 있다고 봅니다.

예시로 나온 것들도 흥미롭습니다.

- HWPX
- 공공기관 양식
- 의약품 안내
- 범정부오피스 자동화
- 한국형 스킬(K-Skill)

왜 중요하냐면,
현업에서는 “멋진 초안”보다 **제출 가능한 결과물**이 훨씬 더 중요하기 때문입니다.

즉 AI의 진짜 가치는 자유롭게 멋진 문장을 쓰는 데만 있는 것이 아니라,
오히려 **정해진 양식, 엄격한 구조, 실수하면 안 되는 문서 환경**에서 더 크게 드러날 수 있다는 뜻입니다.

이 부분은 특히 한국 실무 환경에서는 더 현실적인 포인트처럼 느껴졌습니다.

### 7. 로컬 AI와 개인 개발환경 재구성은 결국 ‘내 작업실을 바꾸는 일’이다
마지막 큰 흐름으로 나온 로컬 AI와 개발환경 이야기도 결국 같은 방향을 가리킵니다.

핵심은 새 모델을 구경하는 게 아니라,
내 장비와 개발환경 자체를 AI 친화적으로 다시 짜는 것입니다.

예시로 나온:
- LM Studio
- mq-dir
- Winget TUI
- DOT Studio
- zero-native

같은 사례는, AI 활용이 더 이상 웹 서비스 소비에서 끝나지 않고,
**내 로컬 환경과 도구 체계 전체를 다시 설계하는 문제**로 넘어간다는 점을 보여줍니다.

즉 이제는 “어떤 AI를 쓸까?” 다음 질문으로,
“내 작업실을 AI가 일하기 좋은 구조로 어떻게 바꿀까?”가 따라붙습니다.

### 8. 이 글의 진짜 가치는 60개 링크보다 ‘읽는 순서’를 만들어 준다는 데 있다
솔직히 60개라는 숫자만 보면 부담스럽습니다.
다 읽으라는 뜻으로 받아들이면 금방 지칩니다.

그런데 이 글의 좋은 점은,
그 많은 자료를 **그대로 들이붓지 않고 구조를 먼저 보여준다**는 데 있습니다.

글이 제안하는 학습 순서도 꽤 실용적이었습니다.

1. 먼저 코딩 에이전트와 목표 기반 작업 감각 잡기
2. 그다음 개인 런타임과 지식 시스템 붙이기
3. 마지막으로 공공 문서와 로컬 환경으로 내려오기

이 순서는 단순히 자료 정렬이 아니라,
**지금 AI를 공부하는 사람이 어디서부터 감을 잡아야 하는지**를 알려주는 지도처럼 느껴졌습니다.

그래서 이 글은 링크 아카이브이기도 하지만,
그보다 더 중요하게는 **학습 경로 설계 문서**처럼 읽혔습니다.

## 아직 헷갈리는 것
- 이런 흐름이 정말 빠르게 표준이 될지, 아니면 일부 파워유저 중심으로 오래 머물지는 더 지켜봐야 할 것 같습니다.
- 모델 성능 차이가 줄어든다고 해도, 여전히 특정 작업에서는 모델 선택이 결정적인 순간이 있을 수 있어서 시스템과 모델의 비중이 어떻게 재조정될지는 더 봐야 합니다.
- MCP, 런타임, 개인 지식 시스템, 문서 자동화가 결국 하나의 통합 스택으로 묶일지, 아니면 각각 따로 발전할지도 아직은 불분명합니다.

## 나중에 더 볼 것
- Codex `/goal`과 AGENTS.md를 실제 작업 흐름에서 더 깊게 보기
- Hermes 계열이 개인 런타임으로 얼마나 안정적으로 자리 잡는지 계속 보기
- MCP 기반 브라우저/도구 연결이 표준처럼 굳어지는지 추적하기
- HWPX와 공공 문서 자동화가 실제 현장에서 어떻게 쓰이는지 사례 더 모으기
- 개인 블로그/노트 시스템도 나중에 AI 재사용 구조까지 염두에 두고 설계해 보기

## 작업 메모
이번 글을 읽고 가장 크게 남은 건,
AI 공부를 계속 “모델 뉴스 따라가기”로만 하면 금방 피곤해질 수 있다는 점이었습니다.

반대로 더 오래 남는 공부는 아마 이런 쪽일 것 같습니다.

- 목표를 어떻게 주는가
- AI가 어떤 도구를 만질 수 있는가
- 작업 흔적과 문서를 어떻게 남기는가
- 검수 가능한 결과물로 어떻게 닫는가

결국 좋은 모델을 찾는 일보다,
**좋은 작업 시스템을 만드는 일**이 더 중요해지는 흐름입니다.

그래서 이번 커뮤니티 학습 글은 단순히 “요즘 뭐가 뜨는가”를 정리한 글이 아니라,
앞으로 무엇을 기준으로 공부할지 다시 잡게 만드는 글로 남겨둘 만했습니다.
