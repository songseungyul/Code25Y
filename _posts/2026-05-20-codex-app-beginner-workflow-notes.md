## 한 줄 요약
이 영상에서 가장 크게 남은 포인트는 **Codex는 프롬프트 한 방으로 결과물을 뽑는 도구라기보다, 프로젝트 폴더와 기준 문서를 세팅해 두고 반복 작업을 맡기는 쪽에 더 잘 맞는 도구**라는 점이었습니다.

## 왜 이 영상을 남겨두나
ChatGPT나 Gemini는 이미 익숙한데, Codex는 이름만 들어 봤지 **언제 더 유리한지**가 흐릿했습니다.

이 영상은 그 차이를 꽤 실무적으로 보여 줍니다.

- 채팅창보다 **프로젝트 폴더**가 중심이라는 점
- 좋은 결과물보다 먼저 **작업 환경**을 설계해야 한다는 점
- 생성보다 **계획 수립과 검수**가 더 중요하다는 점
- 반복 업무는 **문서 구조·스킬·메모리**로 굳혀야 한다는 점

특히 저는 "긴 프롬프트 하나"보다 **`AGENTS.md` 같은 작업 규칙 문서, 디자인 가이드, 비즈니스 컨텍스트, 덱 브리프를 쌓아 가는 방식**이 인상적이었습니다. 나중에 비슷한 자동화 흐름을 만들 때 다시 보기 좋은 메모라서 남깁니다.

## 공부한 것

### 1. Codex의 기본 단위는 대화보다 프로젝트 폴더에 가깝다
영상은 Codex를 단순히 "OpenAI 버전 클로드 코드"라고 소개하지만, 실제로 더 중요한 설명은 그 다음에 나옵니다.

핵심은 Codex가 **작업 폴더를 중심으로 문서, 산출물, 검수 결과를 함께 관리하는 흐름**에 강하다는 점입니다.

영상에서는 예시로 이런 구조를 먼저 만듭니다.

- 프로젝트 루트 폴더
- `AGENTS.md` 또는 작업 규칙 문서
- `design.md`
- `business-context.md`
- `deck-brief.md`
- 산출물 폴더

이 구조가 중요한 이유는, 이후 Codex가 매번 새로 설명을 듣는 게 아니라 **폴더 안에 쌓인 문맥을 참조해서 일한다**는 점 때문입니다.

OpenAI 공식 문서에서도 Codex는 단순 답변 도구가 아니라 **코드를 읽고, 수정하고, 실행하는 에이전트**로 설명됩니다. 특히 Codex 문서에는 App / CLI / IDE / Web / AGENTS.md / Hooks / Skills 같은 항목이 따로 보이는데, 이것만 봐도 설계 철학이 "한 번 답변하고 끝"보다는 **작업 환경 속에서 움직이는 에이전트**에 더 가깝다는 걸 알 수 있습니다.

### 2. 좋은 결과물은 좋은 프롬프트보다 좋은 작업 환경에서 더 잘 나온다
이 영상이 반복해서 강조하는 건 **프롬프트 하나를 잘 쓰는 것보다, Codex가 참조할 기준 문서를 잘 만들어 두는 게 더 중요하다**는 점입니다.

예를 들어 PPT 하나를 만들더라도 그냥

> AI 자동화 컨설팅 제안서 PPT 만들어줘

라고 하면 결과물은 나올 수 있습니다.

하지만 영상은 그 전에 다음을 먼저 세팅합니다.

- 디자인 기준
- 비즈니스 배경
- 고객 문제 정의
- 슬라이드 흐름
- 성공 기준
- 멈춰야 할 기준

이 접근이 인상적이었던 이유는, 결국 업무 자동화에서 중요한 건 **모델이 무엇을 해야 하는지보다 무엇을 참조해야 하는지**라는 걸 잘 보여 주기 때문입니다.

저는 이 부분을 보면서 "프롬프트 엔지니어링"을 너무 좁게 이해하고 있었다는 생각이 들었습니다. 실제 업무에서는 말 잘 거는 기술보다, **모델이 일할 문서 환경을 잘 깔아 주는 기술**이 더 중요할 수 있습니다.

### 3. 생성 전에 계획을 받는 습관이 비용과 품질을 같이 잡아 준다
영상에서 가장 실무적인 팁은 **바로 결과물을 만들지 말고 먼저 Plan 모드로 접근 계획을 받으라**는 부분이었습니다.

이 흐름은 특히 무거운 작업에서 유용해 보였습니다.

- 제안서 작성
- 리서치 기반 문서 작성
- 슬라이드 덱 생성
- 이미지 포함 산출물 제작
- 검수 리포트 생성

이런 작업은 잘못 시작하면 시간도 오래 걸리고 사용량도 많이 먹는데, 먼저 계획을 보면

- 어떤 자료를 참고할지
- 어떤 슬라이드 구조로 갈지
- 어디서 사용자 확인이 필요한지
- 어떤 기준으로 성공 여부를 판단할지

를 미리 정렬할 수 있습니다.

즉, **생성 단계보다 설계 단계에서 품질을 끌어올리는 방식**입니다. 이건 Codex뿐 아니라 다른 에이전트 도구를 쓸 때도 거의 그대로 적용할 수 있는 원칙처럼 느껴졌습니다.

### 4. 반복 업무는 메모리·스킬·훅으로 굳혀야 진짜 자동화가 된다
후반부에서 가장 유용했던 건 "한 번 잘 되었다"에서 끝나지 않고, 그걸 **반복 가능한 시스템으로 바꾸는 방법**까지 짚어 준 점입니다.

영상 기준 핵심 레버는 세 가지였습니다.

- **메모리**: 이전 작업 맥락을 재사용
- **스킬**: 반복 작업을 재사용 가능한 절차로 저장
- **훅**: 특정 검증/체크를 자동으로 강제

OpenAI 공식 문서에서도 Codex 관련 설정 항목에 **Memories, Hooks, AGENTS.md, Skills**가 따로 잡혀 있습니다. 그래서 이 영상의 워크플로는 단순 개인 팁이라기보다, Codex가 원래 지향하는 사용 방식과도 꽤 맞닿아 있다고 볼 수 있습니다.

특히 훅 이야기가 좋았습니다. 단순히 "결과물 검수해 줘"라고 매번 말하는 게 아니라, **어떤 단계에서 어떤 검사를 반드시 수행하도록 구조화**하는 쪽이 훨씬 시스템적이기 때문입니다.

### 5. 검수는 부가 작업이 아니라 생성 파이프라인의 일부다
영상에서 제안서 예시를 만드는 과정은 대략 이렇게 흘러갑니다.

1. 기준 문서 세팅
2. 계획 수립
3. 리서치
4. 슬라이드 생성
5. 재검수
6. 파워포인트 실행 확인
7. 최종 정리

여기서 중요한 건 검수가 마지막 덤이 아니라, **산출물 생성 파이프라인 안에 포함된 단계**라는 점입니다.

특히 인상 깊었던 부분은 다음입니다.

- 장표 렌더 확인
- 수치/사실 관계 재검토
- 슬라이드별 수정 포인트 점검
- 최종 파일을 실제 프레젠테이션 뷰로 넘기며 확인

영상에서는 컴퓨터 사용 기능으로 PowerPoint를 열어 최종 검수하는 흐름까지 보여 주는데, 이건 아주 강력해 보이면서도 동시에 **검수 없는 자동화는 결국 사람에게 다시 비용을 돌린다**는 사실을 보여 줍니다.

즉, 자동화의 핵심은 빨리 만드는 게 아니라 **검수 비용까지 줄이는 것**에 더 가깝습니다.

### 6. 이 영상의 진짜 메시지는 “도구 추천”보다 “운영 방식 전환”에 가깝다
표면적으로는 Codex 튜토리얼 영상이지만, 실제로 더 크게 남는 건 도구 자체보다 **업무를 에이전트에게 맡기는 방식**입니다.

제가 이 영상을 보고 다시 적어 둔 문장은 이겁니다.

> 좋은 자동화는 프롬프트에서 시작하지 않고, 폴더 구조·기준 문서·계획·검수 루프에서 시작한다.

결국 Codex라는 이름보다 더 중요한 건,

- 작업을 프로젝트 단위로 본다
- 기준 문서를 먼저 만든다
- 계획을 먼저 받고 시작한다
- 결과물은 반드시 검수 루프를 거친다
- 자주 하는 일은 스킬과 훅으로 굳힌다

라는 사고방식입니다.

이건 앞으로 다른 에이전트 툴을 보더라도 계속 가져갈 만한 프레임으로 느껴졌습니다.

## 내가 가져갈 포인트
- Codex는 **채팅 도구라기보다 프로젝트 폴더를 중심으로 움직이는 에이전트**로 보는 편이 맞다.
- 결과물 품질은 긴 프롬프트보다 **기준 문서 구조**에 더 크게 좌우될 수 있다.
- 무거운 작업일수록 **Plan → 생성 → 검수** 순서가 중요하다.
- 반복 업무는 한 번 잘하는 것보다 **메모리·스킬·훅으로 고정하는 것**이 더 중요하다.
- 자동화의 진짜 절감 포인트는 생성 속도보다 **검수 비용 감소**에 있다.

## 아직 헷갈리는 것
- 영상에서 보여 준 App 중심 워크플로와 CLI 중심 워크플로 중, 실제 장기 생산성은 어느 쪽이 더 높은지 더 써 봐야 알 것 같습니다.
- 컴퓨터 사용 기능, 플러그인, 메모리 같은 기능이 플랜/OS/권한에 따라 어디까지 열리는지는 사용자 환경별 차이가 있을 수 있습니다.
- `AGENTS.md`, 훅, 스킬을 어느 수준까지 세분화해야 관리가 쉬워지는지는 실제 반복 업무를 몇 번 돌려 봐야 감이 잡힐 것 같습니다.

## 나중에 더 볼 것
- OpenAI Codex 공식 문서에서 **AGENTS.md / Hooks / Skills / Memories** 섹션 따로 읽기
- 제안서 말고 **리서치 정리, 문서 초안, 검수 자동화**에도 같은 구조가 잘 먹히는지 시험해 보기
- 프로젝트 폴더에 넣을 기준 문서 템플릿을 내 작업 방식에 맞게 최소 세트로 정리해 보기
- 반복 업무 하나를 골라 실제로 **스킬 + 훅**까지 묶어 보는 실험 해 보기

## 작업 메모
이 영상은 튜토리얼 형식이라 설명이 빠르고 예시가 많아서, 그대로 따라 하면 바로 뭔가 될 것처럼 느껴집니다.

하지만 다시 정리해 보니, 영상의 핵심은 "Codex가 대단하다"보다 아래에 더 가까웠습니다.

1. 작업 폴더를 만든다  
2. 기준 문서를 만든다  
3. 계획을 먼저 본다  
4. 생성 후 검수 루프를 돈다  
5. 반복되면 메모리·스킬·훅으로 굳힌다

이 다섯 줄만 남겨도 영상의 실질적인 가치는 거의 보존되는 것 같습니다.

그리고 이 흐름은 Codex 전용 팁이라기보다, 앞으로 에이전트 기반 업무 자동화를 공부할 때 **계속 재사용할 수 있는 운영 원칙**으로 남길 만했습니다.
