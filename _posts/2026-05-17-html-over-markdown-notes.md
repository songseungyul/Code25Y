## 한 줄 요약
이 영상의 핵심은 **AI가 만든 긴 결과물을 사람에게 정말 읽히게 하려면, 기본 출력 형식을 마크다운에서 HTML 쪽으로 다시 생각해야 한다**는 점이다. 단순히 보기 좋게 꾸미자는 얘기가 아니라, **사람이 AI와 같은 판단 루프 안에 계속 남아 있으려면 더 읽히는 형식이 필요하다**는 주장으로 들렸다.

## 왜 이 영상을 남겨두나
요즘 AI에게 시켜서 문서를 만들면, 결과물 자체는 빨리 나온다. 문제는 그다음이다.

- 정말 처음부터 끝까지 읽는가
- 읽더라도 비교와 판단이 쉬운가
- 동료와 공유하고 피드백받기 쉬운가
- 다시 AI에게 수정 지시를 넣기 좋은가

이 영상은 바로 이 지점을 찌른다. “마크다운은 익숙한데, 그래서 계속 기본값으로 쓰고 있는 것뿐 아닌가?”라는 질문을 던진다. 특히 **긴 MD 파일을 안 읽게 되는 문제를 사람의 게으름이 아니라 형식의 한계로 본다**는 점이 인상적이었다.

## 공부한 것

### 1. 문제의 출발점은 “AI가 만든 문서를 사람이 끝까지 안 읽는다”는 데 있다
영상은 꽤 직설적으로 시작한다. Claude Code 팀 엔지니어가 스스로 **100줄이 넘는 마크다운은 본인도 잘 안 읽고, 다른 사람에게 읽히기도 어렵다**고 말했다는 것이다.

이 포인트가 중요한 이유는, 우리가 AI 산출물의 품질을 보통 “얼마나 많이 써줬나”, “얼마나 정리돼 보이나”로 착각하기 쉽기 때문이다. 하지만 실제로는:

- 길기만 한 문서보다
- 빨리 훑어볼 수 있고
- 구조가 눈에 들어오고
- 필요한 지점에서 직접 조작할 수 있는 문서

가 훨씬 실용적일 수 있다.

결국 문제는 “AI가 잘 썼는가”보다 **사람이 그 결과물을 소비할 수 있는가**에 가까워 보였다.

### 2. 영상이 말하는 해결책은 생각보다 단순하다: “HTML로 만들어줘” 한 줄
영상에서 가장 반복되는 메시지는 이거다.

> 프롬프트 끝에 “HTML로 만들어줘” 한 줄만 붙여 봐라.

복잡한 새 툴을 설치하거나 별도 스킬을 만들라는 얘기가 아니다. 출력 포맷만 바꾸면 된다는 주장이다. 이 단순함이 오히려 인상적이었다.

예를 들면:

- 스펙 정리해줘 → **스펙을 HTML로 정리해줘**
- 리팩토링 계획 짜줘 → **리팩토링 계획을 HTML로 짜줘**
- 코드베이스 설명해줘 → **다이어그램 포함 HTML 설명서로 만들어줘**

이 정도 변화만으로도 결과물의 사용성이 크게 달라질 수 있다는 게 영상의 핵심 제안이다.

### 3. 왜 HTML이 더 낫다고 보나: 정보 밀도와 시각적 명확성
영상은 HTML 쪽의 장점을 크게 두 갈래로 본다.

첫째는 **표현력**이다.

마크다운은 익숙하지만 표현 가능한 요소가 꽤 제한적이다.

- 색상 강조
- 탭 구조
- 카드형 비교
- SVG 다이어그램
- 일러스트
- 버튼
- 애니메이션
- 슬라이더 같은 인터랙션

이런 것들은 HTML 쪽이 훨씬 유리하다.

둘째는 **읽힘성**이다.

긴 마크다운은 스크롤만 길어지고, 사람은 중간부터 대충 훑고 닫아 버리기 쉽다. 반면 HTML은:

- 섹션을 더 명확하게 나눌 수 있고
- 모바일에서 보기 좋게 구성할 수 있고
- 비교 항목을 카드나 그리드로 보여줄 수 있고
- 다이어그램이나 그림으로 문맥을 압축할 수 있다.

즉 영상은 HTML을 단순한 꾸밈 포맷이 아니라, **AI 산출물을 사람이 읽기 좋은 형태로 재구성하는 그릇**으로 본다.

### 4. 이 영상이 특히 강조하는 건 “양방향 인터랙션”이다
여기서부터는 단순 문서가 아니라 **미니 도구**에 가까워진다.

영상이 든 예시는 꽤 구체적이다.

- 애니메이션 속도 슬라이더
- 색상 조절 슬라이더
- 두께 조절 슬라이더
- 마음에 드는 값을 다시 프롬프트로 복사하는 버튼

이런 구조가 있으면, 사람은 결과물을 읽기만 하는 게 아니라 직접 만져볼 수 있다. 그리고 그 결과를 다시 AI에게 되먹임할 수 있다.

이 지점이 중요한 이유는, AI를 더 잘 쓰는 문제를 **정답 요청**에서 **상호작용 설계**로 옮겨 놓기 때문이다. 그냥 요약을 읽고 끝나는 게 아니라, 결과물을 조정하면서 다음 지시로 이어지는 루프를 만들 수 있다.

### 5. 영상 속 Karpathy 메시지는 “출력의 진화 단계”라는 프레임으로 읽힌다
영상은 이 흐름을 단순 팁으로 끝내지 않고, Karpathy의 코멘트를 빌려 더 크게 해석한다.

대략 이런 구도다.

1. 텍스트만 있는 출력
2. 기본 서식이 있는 마크다운 출력
3. 그래픽, 레이아웃, 인터랙션을 담는 HTML 출력
4. 더 나아가면 인터랙티브 영상 같은 단계

핵심은 “AI가 사람에게 보여 주는 출력도 점점 더 시각적이고 상호작용적인 방향으로 진화할 것”이라는 관점이다.

이 프레임이 흥미로운 이유는, 출력 형식을 단순 취향 문제가 아니라 **AI 시대 인터페이스의 변화**로 보게 만들기 때문이다. 예전에는 텍스트만으로도 충분했을지 몰라도, 지금은 AI가 길고 복잡한 결과물을 대량으로 만들기 때문에, 사람이 받아들이는 포맷도 달라질 수밖에 없다는 뜻이다.

### 6. 결국 포인트는 “사람이 AI의 판단을 검토하는 루프에 계속 남아 있느냐”다
영상 후반부에서 가장 중요하게 느껴진 문장은 이것에 가까웠다.

- 마크다운 결과물을 점점 안 읽게 되면
- 결국 AI가 정리한 결론을 그냥 따라가게 되고
- 사람은 검토자 자리에서 밀려난다.

반대로 HTML처럼 더 읽히고 더 만질 수 있는 형식이면:

- 결과물을 더 끝까지 보게 되고
- 중간에 직접 비교하고 판단하게 되고
- 필요한 수정 포인트를 다시 AI에게 던지게 된다.

즉 형식의 문제는 미감의 문제가 아니라, **판단 권한이 사람에게 남는가**의 문제로 이어진다. 이 해석은 꽤 오래 남을 것 같다.

### 7. 영상이 제시한 다섯 가지 활용 예시는 모두 “문서”보다 “즉석 작업면”에 가깝다
영상에서 소개한 활용 시나리오는 다음 다섯 가지였다.

1. **스펙/계획 비교 페이지**  
   여러 옵션을 그리드로 한 번에 비교하고, 트레이드오프를 함께 본다.

2. **코드 리뷰 / PR 설명서**  
   diff 옆에 주석과 위험도 색상을 붙여 리뷰 가독성을 높인다.

3. **디자인 프로토타입**  
   슬라이더를 움직이며 파라미터를 직접 맞춰 본다.

4. **리포트 / 리서치 / 인수인계 문서**  
   SVG 다이어그램까지 포함한 발표 자료급 설명서를 만든다.

5. **1회성 커스텀 에디터**  
   예를 들어 티켓 우선순위를 드래그로 재배치하고, 결과를 다시 텍스트로 추출한다.

이 다섯 가지를 묶어 보면 공통점이 뚜렷하다. HTML은 정적 결과물을 예쁘게 만드는 데만 쓰이는 것이 아니라, **그 순간의 판단 작업을 돕는 임시 인터페이스**로도 쓸 수 있다는 것이다.

### 8. 단점도 분명하다: 토큰 비용과 버전 관리 노이즈
영상은 HTML이 무조건 좋다고만 말하지는 않는다. 단점도 분명히 언급한다.

- 마크다운보다 토큰을 더 많이 쓸 수 있다
- 응답 시간이 길어질 수 있다
- HTML은 diff가 지저분해질 수 있다

이건 실제로 중요한 단점이다. 특히 깃으로 버전 관리하는 문서라면, 작은 수정도 줄바꿈과 구조 변경 때문에 더 복잡하게 보일 수 있다.

다만 영상의 결론은 명확하다. **토큰을 조금 더 쓰는 손해보다, 사람이 결과물을 아예 안 읽는 손해가 더 크다**는 것이다. 이 판단에는 꽤 설득력이 있었다.

## 내가 가져갈 포인트
이 영상을 보고 남긴 핵심은 세 가지다.

1. **AI 산출물의 품질은 내용만이 아니라 소비 형식까지 포함해서 봐야 한다.**  
   잘 쓴 문서와 잘 읽히는 문서는 같지 않을 수 있다.

2. **마크다운은 익숙한 기본값일 뿐, 항상 최적값은 아닐 수 있다.**  
   특히 비교, 시각화, 조작이 중요한 작업일수록 HTML 쪽이 더 자연스럽다.

3. **AI와 사람의 협업 루프를 유지하려면, 사람이 끝까지 읽고 만질 수 있는 결과물이 중요하다.**  
   보기 좋은 문서가 목적이 아니라, 판단 가능한 인터페이스가 목적이라는 점이 남았다.

## 아직 헷갈리는 것
- 모든 종류의 문서에서 HTML이 더 낫다고 볼 수 있는지는 더 실험이 필요하다.
- 팀 협업에서 HTML 산출물을 어디에 저장하고 어떻게 버전 관리할지가 별도 과제로 남는다.
- 마크다운이 여전히 더 좋은 경우, 예를 들면 장기 보관용 텍스트 문서나 diff 중심 리뷰 문서의 경계도 더 따져보고 싶다.

## 나중에 더 볼 것
- 실제로 같은 요청을 마크다운/HTML 두 형식으로 각각 뽑아 비교해 보기
- PR 설명서나 리팩토링 계획을 HTML로 만들어 보는 실험 해 보기
- HTML 결과물에서 다시 AI로 되먹임하는 버튼/폼 구조를 어떻게 잘 설계할지 공부하기

## 작업 메모
이 영상은 단순히 “HTML이 더 화려하다”는 얘기가 아니다. 오히려 **AI가 더 많은 문서를 더 빠르게 생산하는 시대에, 사람의 읽기와 판단을 다시 중심에 놓는 형식이 무엇인가**를 묻는 영상에 가깝다.

그래서 이 메모를 남겨둔다. 앞으로 AI를 더 자주 쓸수록 중요한 건 “무엇을 시킬까”만이 아니라 **어떤 형태로 돌려받아야 내가 정말 읽고 판단할 수 있을까**일 가능성이 크기 때문이다.
