## 한 줄 요약
이 발표의 핵심은 **코딩 에이전트의 성능 차이는 결국 모델 자체보다, 그 모델이 어떤 컨텍스트를 읽고 어떤 도구를 쓰며 어떤 검증 루프 안에서 일하도록 설계되어 있는가에서 크게 갈린다**는 주장입니다.

## 왜 이 영상을 남겨두나
요즘 AI 코딩 도구를 쓰다 보면 비슷한 모델을 써도 결과 차이가 너무 크게 납니다.

- 누구는 MVP 수준에서 자꾸 멈추고
- 누구는 실제 운영 가능한 결과물까지 밀고 가고
- 누구는 빠르지만 점점 코드가 흐트러지고
- 누구는 속도는 느려도 결과가 안정적입니다.

이 영상이 좋았던 이유는 이 차이를 단순히 "프롬프트를 잘 쓰는 사람"과 "못 쓰는 사람"의 차이로 설명하지 않고, **에이전트가 일하는 전체 환경을 어떻게 설계했는가**의 문제로 끌어올려 설명하기 때문입니다.

즉 이 발표는 특정 도구 사용 팁이라기보다, **AI 시대의 개발자가 무엇을 설계해야 하는가**를 다시 묻는 메모로 남겨둘 가치가 있었습니다.

## 공부한 것

### 1. 하네스 엔지니어링은 ‘모델 바깥의 모든 것’을 설계하는 일에 가깝다
발표에서는 하네스를 아주 넓게 정의합니다. 단순히 모델에 붙는 부가 기능이 아니라, **모델을 둘러싼 작업 환경 전체**를 뜻한다는 식입니다.

여기서 말하는 하네스의 구성 요소는 대략 이런 것들입니다.

- 어떤 목표를 주는가
- 어떤 제약과 가드레일을 주는가
- 어떤 컨텍스트를 처음부터 넣고, 무엇은 필요할 때만 불러오게 할 것인가
- 어떤 도구를 허용하거나 금지할 것인가
- 작업 완료 여부를 무엇으로 검증할 것인가
- 사람이 언제 개입하고 어디서 승인할 것인가

이 정의가 좋았던 이유는, AI 코딩을 더 이상 "모델에게 코드를 시키는 일"로만 보지 않게 만들기 때문입니다. 이제 더 중요한 건 **에이전트가 흔들리지 않게 일하는 구조를 설계하는 일**이라는 메시지에 가깝습니다.

### 2. 핵심은 ‘요청 → 생성’이 아니라 ‘목표 → 검증 → 재시도’ 루프다
발표 초반부에서 가장 선명했던 부분은, 기존의 AI 코딩 흐름이 사람을 너무 지치게 만든다는 지적이었습니다.

예전 방식은 대체로 이렇습니다.

1. 사람이 요청을 쓴다.
2. 모델이 코드를 낸다.
3. 사람이 직접 확인한다.
4. 문제가 있으면 다시 설명한다.
5. 이 과정을 끝없이 반복한다.

이 구조에서는 검증 루프가 사람 머릿속에만 있습니다. 그래서 반복될수록 사람이 피로해집니다.

발표자는 이걸 바꿔서 다음처럼 봐야 한다고 말합니다.

1. 먼저 최종 목표를 준다.
2. 중간 단계마다 만족해야 할 검증 기준을 둔다.
3. 에이전트가 탐색·수정·생성을 반복한다.
4. 검증에 실패하면 다시 루프로 들어간다.
5. 목표를 잃거나 애매해지면 사람에게 되묻는다.

이 관점에서 개발자는 직접 손을 많이 움직이는 사람이기보다, **작업 루프와 승인 루프를 설계하는 관리자**에 가까워집니다.

### 3. 발표는 성능 격차를 만드는 세 축으로 ‘컨텍스트·도구·평가’를 강조한다
이 발표의 골격은 꽤 단순합니다. 같은 모델을 써도 결과가 달라지는 이유를 세 가지 축으로 설명합니다.

- **컨텍스트 엔지니어링**
- **도구와 실행 환경 설계**
- **이밸루에이션(평가) 기반 개선 루프**

이 세 가지를 따로 떼어 외우기보다, 저는 이렇게 읽는 편이 더 남았습니다.

- 컨텍스트는 에이전트가 **무엇을 알고 시작하는가**의 문제
- 도구는 에이전트가 **무엇을 할 수 있는가**의 문제
- 평가는 에이전트가 **잘했는지 어떻게 판정할 것인가**의 문제

결국 세 가지가 합쳐져야 작업이 안정됩니다.

### 4. 컨텍스트는 많이 넣는 것이 아니라, ‘처음부터 필요한 것’과 ‘필요할 때 읽을 것’을 나누는 게 중요하다
발표자는 컨텍스트를 예산으로 봐야 한다고 강조합니다. 컨텍스트 윈도우가 커졌다고 해서 무한정 정보를 넣는 것이 좋은 게 아니라, **정말 가치가 높은 정보만 엄선해야 한다**는 말입니다.

여기서 특히 좋았던 구분은 두 가지였습니다.

#### 처음부터 바로 주입해야 하는 것
- 작업 목표
- 제약 사항
- 코딩 컨벤션
- 최근 아키텍처 변경 사항
- 절대 하면 안 되는 행동
- 보안 관련 주의점

#### 필요할 때만 도구로 가져오면 되는 것
- 세부 코드 내용
- 대용량 로그
- 외부 문서와 API 레퍼런스
- 테스트 실행 결과 전체
- 긴 에러 트레이스 원문

이 구분이 중요한 이유는, 긴 로그나 테스트 출력은 생각보다 컨텍스트를 심하게 오염시키기 때문입니다. 발표도 이 점을 반복해서 짚습니다. 토큰이 많아질수록 정확도가 자동으로 올라가는 게 아니라, 오히려 **잡음과 오류 가능성도 함께 커진다**는 쪽입니다.

그래서 핵심은 "무엇을 더 넣을까"보다 **무엇을 처음부터 넣지 않을까**에 더 가까워 보였습니다.

### 5. 문서는 ‘다 넣는 백과사전’이 아니라, 필요한 문서로 들어가기 위한 인덱스여야 한다
발표 중간에는 문서 구조 설계 이야기가 나옵니다. 특히 루트에 두는 `AGENTS.md` 또는 유사한 규칙 문서를 **프로젝트의 목차**처럼 써야 한다는 설명이 인상적이었습니다.

요점은 이런 식입니다.

- 루트 문서에는 아주 자세한 내용을 다 넣지 않는다.
- 대신 어디에 어떤 문서가 있는지 길잡이 역할을 하게 만든다.
- 세부 내용은 아키텍처 문서, 제품 스펙 문서, 레퍼런스 문서 등으로 분리한다.
- 실행 계획도 별도 문서로 남겨 회고 가능하게 만든다.

이 부분은 실무적으로 꽤 중요해 보였습니다. 코드 결과물은 Git에 남지만, **왜 그런 계획이 나왔는지**는 세션이 끝나면 쉽게 사라지기 때문입니다.

즉 잘 만든 하네스는 코드만 남기는 게 아니라, **계획과 판단의 흔적도 재사용 가능한 문서로 남기는 구조**를 가져야 합니다.

### 6. 도구 설계에서는 ‘무엇을 줄까’만큼 ‘무엇을 막을까’도 중요하다
발표는 도구와 실행 환경을 꽤 보수적으로 다룹니다. 아무 도구나 다 열어 주는 게 능사가 아니라, 오히려 **도구 오버랩과 잘못된 자유도를 줄이는 것**이 중요하다고 말합니다.

예를 들어 발표자는 기본 웹 검색 대신 더 나은 외부 검색 도구를 쓰기도 하고, 특정 도구는 아예 실행하지 않도록 규칙으로 막는다고 설명합니다.

또 인상적이었던 포인트는 승인 기반 루프에 대한 태도였습니다.

- 처음부터 바이패스 모드를 열어 주지 않는다.
- 사용자가 직접 승인하는 감각을 먼저 익히게 한다.
- 도구 에러 메시지도 반드시 전달되게 한다.
- 가능하면 샌드박스 위에서 먼저 실행해 안전성을 판단한다.

즉 하네스는 생산성을 높이는 장치이면서 동시에, **에이전트가 사고를 덜 치게 만드는 안전장치**이기도 합니다.

### 7. 평가 없는 AI 코딩은 결국 ‘느낌으로 만든 코드’가 되기 쉽다
발표에서 가장 강하게 반복된 메시지는 이 부분이었습니다. **코드베이스는 느낌으로 개선하면 안 된다**는 말입니다.

발표자가 말하는 이밸루에이션은 단순한 감상이 아닙니다.

- 어떤 작업을 어떤 기준으로 평가할지 정의하고
- 여러 번 실행해 보고
- 점검 과정을 기록하고
- 테스트, 린트, UI 검증 같은 형태로 결과를 확인하고
- 그 결과를 바탕으로 다시 개선하는 구조

를 뜻합니다.

여기서 특히 중요한 지점은 **자기 평가의 함정**입니다. 같은 세션에서, 같은 모델이 자기 결과물을 스스로 채점하게 하면 편향이 심하게 생깁니다. 에이전트는 자신이 한 일을 과하게 긍정하는 경향이 있기 때문입니다.

이 발표를 듣고 남는 결론은 꽤 단순합니다.

- 작성자와 검증자를 분리하고
- 가능하면 세션도 분리하고
- 테스트·린트·스크린샷 같은 외부 기준으로 확인하라

즉 "잘 만든 것 같은데?"가 아니라, **무엇을 통과했는가**로 품질을 판단해야 합니다.

### 8. 하네스는 한 번 완성하는 게 아니라, 모델과 함께 계속 바뀌는 개발 환경이다
이 발표가 좋았던 또 하나의 이유는, 하네스를 정답처럼 고정하지 않는다는 점입니다.

발표자는 모델이 바뀌면 많은 하네스가 불필요해지기도 하고, 반대로 새로운 문제 때문에 더 복잡한 보완이 필요해지기도 한다고 말합니다. 즉 오늘 잘 먹히는 구조가 내일도 그대로 정답일 필요는 없습니다.

이건 꽤 현실적인 관점입니다.

- 모델이 좋아지면 과거의 보조 장치가 불필요해질 수 있고
- 새로운 도구가 나오면 기존 흐름이 재편될 수 있고
- 프로젝트 단계가 바뀌면 검증 방식도 달라질 수 있습니다.

그래서 하네스는 정적인 설정이 아니라, **계속 고쳐 나가는 개발 환경 자체**로 보는 편이 더 맞아 보였습니다.

### 9. 발표자는 Claude Code와 Codex를 경쟁 제품보다 역할 분담 파트너에 가깝게 본다
영상 후반부에서는 Claude Code와 Codex 비교가 꽤 긴 비중을 차지합니다. 발표자의 요약은 대략 이런 식입니다.

- **Claude Code**는 코드 작성에 강한 페어 프로그래머에 가깝다.
- **Codex**는 계획 검토, 컨텍스트 탐색, 검증, 루프 오케스트레이션에 더 강하다고 본다.

발표자가 실제로 쓰는 방식도 흥미로웠습니다.

1. 둘 다 같은 작업에 대한 계획을 세운다.
2. Codex가 그 계획을 더 우선적으로 검토·수정한다.
3. 1차 구현은 Claude Code가 맡는다.
4. 1차 검증과 수정은 Codex가 맡는다.
5. 다시 Claude Code가 보고,
6. 최종 검증은 또 Codex가 맡는다.

핵심은 한 도구를 "최종 승자"로 고르는 데 있지 않습니다. 오히려 **같은 모델이 자기 결과를 다시 검증하지 않게 역할을 엇갈리게 배치하는 것**이 더 중요하다는 쪽입니다.

이 부분은 발표자의 운영 방식이기 때문에 모두에게 그대로 정답일 필요는 없지만, 최소한 한 가지는 분명하게 남습니다. AI 코딩 도구를 비교할 때도 단순 벤치마크보다 **어떤 단계에 어떤 역할로 넣을 것인가**를 먼저 봐야 한다는 점입니다.

### 10. 팀 단위로 가면 하네스는 개인 노하우가 아니라 공유 자산이 되어야 한다
발표 후반부에는 팀 운영 이야기도 나옵니다. 여기서 핵심은, 한 사람이 잘 쓰는 프롬프트나 규칙을 자기 로컬 환경에만 숨겨두면 팀 전체의 생산성으로 이어지지 않는다는 점입니다.

그래서 발표자는 다음 같은 방향을 제안합니다.

- 팀 공통 규칙을 별도 레포로 관리한다.
- 프로젝트에는 서브모듈 등으로 연결한다.
- 반복적으로 유효했던 작업 절차는 스킬로 만들어 공유한다.
- 훅과 스크립트는 가능한 코드 수준에서 강제한다.
- 문서로만 남기지 말고, 세션 시작 시 자동 주입되게 만든다.

이건 결국 하네스가 개인 팁을 넘어서 **팀의 운영 지식과 품질 기준을 담는 저장소**가 되어야 한다는 뜻으로 읽혔습니다.

### 11. 발표가 말하는 미래의 개발자는 ‘코드를 많이 치는 사람’보다 ‘에이전트가 일하는 구조를 설계하는 사람’에 더 가깝다
마지막 메시지는 꽤 강했습니다. 발표자는 앞으로 개발자의 직함이 단순히 코드를 작성하는 사람을 뜻하지 않게 될 수 있다고 말합니다.

이 문장을 과장으로만 보긴 어려웠습니다. 영상 전체 논리를 따라가 보면 자연스럽습니다.

- 모델은 점점 평준화되고
- 도구 접근성도 넓어지고
- 단순 구현 속도 차이는 줄어들 수 있지만
- 작업 환경을 어떻게 설계하느냐는 여전히 큰 차이를 만듭니다.

그래서 앞으로 더 중요해질 역량은:

- 반복 업무를 문서화하는 능력
- 규칙과 제약을 구조화하는 능력
- 검증 기준을 명시하는 능력
- 사람과 에이전트의 개입 지점을 설계하는 능력
- 팀 차원의 공유 자산으로 정리하는 능력

같은 것들일 수 있겠다는 생각이 들었습니다.

## 내가 가져갈 포인트
- 하네스 엔지니어링은 프롬프트 요령보다 **에이전트가 일하는 환경 설계**에 더 가깝다.
- 성능 차이는 모델보다도 **컨텍스트·도구·평가 루프**의 설계에서 크게 벌어진다.
- 컨텍스트는 많이 주는 게 아니라, **처음부터 필요한 것과 나중에 읽을 것을 나누는 일**이 중요하다.
- 검증 없는 AI 코딩은 결국 **느낌 기반 개발**로 흐르기 쉽다.
- 같은 모델이 자기 결과를 다시 칭찬하게 두지 말고, **교차 검증 구조**를 만드는 편이 낫다.
- 하네스는 개인 생산성 팁이 아니라, 나중에는 **팀의 운영 자산**이 되어야 한다.

## 아직 헷갈리는 것
- 발표자가 제안한 Claude Code / Codex 역할 분담이 모든 팀과 프로젝트에 그대로 맞는지는 더 실험이 필요합니다.
- 컨텍스트를 얼마나 줄여야 좋은지, 어느 지점부터 오히려 정보 부족이 생기는지는 프로젝트 크기마다 다를 것 같습니다.
- 승인 기반 루프와 바이패스 자동화 사이의 균형도, 속도와 안전성 요구에 따라 꽤 달라질 수 있습니다.
- 평가를 강조하는 방향은 매우 설득력 있었지만, 실제로 어떤 지표를 어디까지 자동화할지는 팀별로 더 구체화가 필요해 보입니다.

## 나중에 더 볼 것
- 지금 쓰는 프로젝트 하나를 골라 `목표 / 제약 / 검증 기준`을 먼저 문서화해 보기
- 긴 로그와 테스트 출력이 정말 컨텍스트를 오염시키는지 체감 비교해 보기
- 같은 작업을 단일 모델 검증과 교차 검증 구조로 각각 돌려 차이 기록해 보기
- 반복적으로 잘 먹힌 프롬프트를 그냥 저장하지 말고, 스킬이나 규칙 문서 형태로 구조화해 보기

## 작업 메모
이 발표를 보며 가장 크게 남은 건, AI 코딩 시대의 경쟁력이 더 이상 "누가 더 빨리 코드를 뽑느냐"에만 있지 않다는 점이었습니다.

오히려 중요한 건 모델을 둘러싼 환경을 얼마나 잘 설계하느냐였습니다. 무엇을 처음부터 읽힐지, 무엇은 필요할 때만 꺼내게 할지, 어떤 도구를 허용할지, 어디서 검증할지, 언제 사람이 개입할지 같은 것들이 쌓여서 결과 품질의 차이를 만든다는 설명이 꽤 설득력 있었습니다.

특히 좋았던 건 하네스를 거창한 비밀 기술처럼 말하지 않았다는 점입니다. 결국 시작점은 복잡한 멀티에이전트 시스템이 아니라, **지금 반복되는 일을 문서화하고 검증 기준을 붙이고 실행 흔적을 남기는 작은 습관**에 더 가까워 보였습니다.
