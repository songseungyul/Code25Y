## 한 줄 요약
이 글의 핵심은 단순히 **Claude Code 플러그인 하나를 소개하는 것**이 아니라, AI가 코드를 너무 쉽게 뱉는 시대일수록 오히려 **무엇을 만들지 먼저 정의하는 흐름**이 더 중요해진다는 점이었습니다.

## 왜 이 글을 남겨두나
AI 코딩 도구를 쓰다 보면 처음에는 감탄하게 됩니다. 요청하면 바로 코드를 만들고, 화면도 금방 나오고, 간단한 프로토타입은 정말 빨리 나옵니다.

그런데 조금만 길게 가면 금방 다른 문제가 생깁니다.

- 처음 말하지 않은 기능이 슬그머니 추가되고
- 빼달라고 하면 다른 기능이 고장 나고
- AI는 늘 “수정했습니다”라고 말하지만
- 진짜로 스펙대로 되었는지는 사람이 다시 확인해야 합니다.

이번 Maily 글은 바로 그 지점을 잘 짚고 있었습니다.

이 글은 “AI가 코딩을 잘한다”보다,
**AI가 사람과 협업 가능한 방식으로 일하게 만들 수 있느냐**를 더 중요하게 봅니다.

## 공부한 것

### 1. 이 글이 진짜로 말하는 문제는 “AI가 코드부터 짜버린다”는 점이다
글에서 가장 먼저 나오는 문제의식은 익숙합니다.

“심플한 텍스트 메모장”을 원했는데,
시간이 지나자 AI가 허락 없이 이미지 첨부, 소셜 공유 같은 기능을 붙이기 시작하고,
그걸 다시 빼려 하면 코드가 이미 꼬여 있어서 다른 데까지 무너진다는 이야기입니다.

이건 단순히 특정 모델이 멍청해서 생기는 문제가 아니라,
**설계도 없이 구현부터 시작하는 워크플로 자체의 문제**처럼 보였습니다.

즉,
- 0에서 1까지 데모를 만드는 속도는 빨라졌지만
- 1에서 100으로 다듬는 과정은 여전히 고통스럽고
- 오히려 AI가 빠를수록 잘못된 방향으로도 빨리 멀어질 수 있다는 뜻입니다.

이 글이 좋았던 이유는, 이 문제를 “프롬프트를 더 잘 쓰면 된다” 수준이 아니라,
**개발 순서를 바꿔야 한다**는 이야기로 끌고 갔기 때문입니다.

### 2. Superpowers의 본질은 플러그인이라기보다 ‘개발 방법론 강제 장치’에 가깝다
Maily 글에서 다룬 Superpowers는 Claude Code 플러그인으로 소개되지만, 공식 README를 같이 보면 성격이 더 분명합니다.

README는 Superpowers를 단순 기능 모음이 아니라,
**coding agents를 위한 complete software development workflow**로 설명합니다.

즉 핵심은 “코드를 잘 짜는 보조도구”가 아니라,
AI가 자동으로 다음 순서를 따르게 만드는 데 있습니다.

- 요구사항을 먼저 묻기
- 스펙을 문서로 만들기
- 구현 계획 세우기
- 구현을 단계적으로 진행하기
- 스펙 준수와 코드 품질을 다시 검토하기

이 관점으로 보면 Superpowers는 기능성 플러그인보다,
**AI용 Spec-Driven Development 습관을 주입하는 프레임워크**에 더 가깝습니다.

### 3. 가장 중요한 변화는 “바로 코딩”에서 “먼저 질문”으로 넘어가는 점이다
글에서 특히 강조된 부분은 `brainstorming` 단계였습니다.

일반적인 AI 코딩 툴은:
- “메모앱 만들어줘”라고 하면
- 거의 반사적으로 코드부터 만들기 시작합니다.

반면 Superpowers는:
- 로컬 저장인지 클라우드인지
- 검색이 필요한지
- 마크다운 지원이 필요한지
- 웹 앱인지 네이티브 앱인지
같은 질문을 먼저 던집니다.

겉으로 보면 조금 답답할 수 있습니다. 당장 코드가 안 나오니까요.

하지만 생각해 보면 이 질문들이야말로 보통 사람이 개발자와 협업할 때 가장 많이 빠뜨리는 부분입니다.

그래서 이 단계의 의미는 단순합니다.

**AI가 코딩하기 전에, 사람이 아직 정의하지 않은 요구사항의 빈칸을 먼저 드러내 준다.**

이건 개발자보다도 오히려 PM이나 기획자 입장에서 더 큰 장점처럼 보였습니다.

### 4. 스펙이 대화로 끝나지 않고 파일로 남는다는 점이 의외로 크다
Maily 글에서 제가 특히 크게 느낀 대목은,
Superpowers가 브레인스토밍 결과를 그냥 대화 컨텍스트 안에 흘려보내지 않고 **문서로 저장**한다는 부분이었습니다.

이건 AI 시대에 생각보다 중요합니다.

왜냐하면 대화형 AI는 본질적으로:
- 세션이 끊기면 맥락이 약해지고
- 예전에 무엇을 합의했는지 흐려지기 쉽고
- 나중에는 “이 기능이 왜 들어갔지?”를 추적하기 어렵기 때문입니다.

그런데 스펙이 문서로 남으면:
- 세션이 바뀌어도 기준점이 유지되고
- 서브에이전트도 같은 문서를 참조하고
- 사람도 결과물을 스펙 기준으로 검증할 수 있습니다.

즉 스펙이 기억이 아니라 **아티팩트**가 됩니다.

이 차이는 꽤 큽니다.
AI를 잘 쓰는 문제를 “기억력이 좋은 조수 고용”이 아니라,
**문서 기반 협업 체계 만들기**로 바꾸기 때문입니다.

### 5. 구현보다 앞에 계획을 두고, 계획보다 앞에 승인을 둔다
글의 흐름을 보면 Superpowers는 대충 이런 순서를 강제합니다.

1. 브레인스토밍
2. 스펙 작성
3. 사용자 승인
4. 구현 계획 작성
5. 다시 승인
6. 구현 시작
7. 별도 검토

공식 README에서도 비슷한 축이 반복됩니다.

- `brainstorming`
- `writing-plans`
- `subagent-driven-development` 또는 실행 단계
- `test-driven-development`
- `requesting-code-review`

이 구조에서 중요한 건 기술 용어 자체보다,
**승인 가능한 중간 단계가 계속 끼어 있다는 점**입니다.

사람이 “잠깐, 이건 내가 원한 게 아닌데”라고 말할 수 있는 시점이 구현 전에 여러 번 생깁니다.

결국 이 방식은 AI를 더 똑똑하게 만드는 게 아니라,
**AI가 틀렸을 때 너무 늦기 전에 멈출 수 있게 만드는 방식**이라고 느껴졌습니다.

### 6. 검증의 우선순위가 ‘코드 품질’보다 ‘스펙 준수’라는 점이 좋았다
Maily 글에서 인상적이었던 문장은 바로 이것이었습니다.

**“코드 품질보다 스펙 준수를 먼저 본다.”**

보통 AI 코딩을 이야기할 때는:
- 코드가 깔끔한가
- 구조가 좋은가
- 리팩터링이 잘 되었나
를 먼저 떠올리기 쉽습니다.

그런데 실제 제품 작업에서는 종종 그보다 먼저 봐야 할 게 있습니다.

- 애초에 요구한 기능이 맞나
- 범위를 넘지 않았나
- 빠진 요구사항은 없나
- 원래 합의와 충돌하지 않나

코드가 예쁘더라도 스펙을 어기면 결국 다시 갈아엎어야 합니다.

그래서 별도 리뷰어 서브에이전트가 먼저 **스펙 준수 여부**를 보고,
그 다음에 코드 품질을 보는 구조는 꽤 현실적이라고 느껴졌습니다.

특히 AI가 스스로 자기 결과물에 관대해지기 쉬운 상황에서는,
구현 세션과 검토 세션을 분리하는 것 자체가 큰 의미가 있어 보였습니다.

### 7. Visual Companion은 ‘AI와 같이 기획한다’는 감각을 강화한다
글 중간에는 Visual Companion 이야기도 나옵니다.

이 기능은 AI가 필요하다고 판단하면:
- 로컬 HTTP 서버를 띄우고
- HTML 기반 목업이나 비교안을 만들고
- 사용자가 브라우저에서 보고 선택하게 하는 식으로 동작합니다.

이 부분이 흥미로웠던 이유는,
단순히 결과 화면을 보여줘서가 아닙니다.

이 기능은 **텍스트 설명만으로는 애매한 기획 결정을 시각적 선택으로 바꿔 준다**는 데 의미가 있습니다.

예를 들어:
- 레이아웃 안 비교
- UI 목업 확인
- 흐름도/다이어그램 확인

같은 것들은 말로 길게 주고받는 것보다,
직접 보고 “이쪽이 낫다”고 선택하는 게 훨씬 빠를 때가 많습니다.

그래서 이 기능은 “AI가 더 빨리 만든다”보다는,
**AI와 사람이 같은 화면을 보며 더 잘 협의한다**는 철학을 잘 보여주는 기능처럼 읽혔습니다.

### 8. 결국 이 글은 ‘생산성 향상’보다 ‘협업 방식 변화’에 대한 글이다
표면적으로는 Claude Code 플러그인 사용기처럼 보이지만,
정작 남는 메시지는 따로 있었습니다.

이 글이 말하는 핵심은:

- AI가 코드 생성 속도를 높여 준다
보다
- AI가 사람과 협업하는 단계를 더 잘 지키게 만든다
에 가까웠습니다.

그리고 그 변화는 결국 이런 문장으로 요약됩니다.

**AI가 코드를 짜는 게 쉬워질수록, 진짜 병목은 무엇을 만들지 정확히 정의하는 일로 옮겨간다.**

저는 이 문장이 이 글의 중심이라고 느꼈습니다.

## 아직 헷갈리는 것
- 실제로 이 방식이 작은 개인 프로젝트에서도 항상 이득인지, 아니면 어느 정도 복잡한 작업부터 진가가 나오는지는 더 봐야 할 것 같습니다.
- 토큰 사용량과 시간 증가가 꽤 큰 단점으로 보이는데, 그 비용을 감수할 만큼의 안정성을 언제부터 체감하는지도 궁금합니다.
- Superpowers가 강제하는 프로세스가 팀과 프로젝트 종류에 따라 얼마나 유연하게 조절되는지는 실제 사용 경험이 더 필요합니다.

## 나중에 더 볼 것
- Superpowers를 Claude Code 말고 다른 에이전트 환경에서도 비슷하게 쓸 수 있는지 확인해 보기
- spec-driven development가 AI 코딩 워크플로에서 어디까지 표준이 될 수 있을지 보기
- 문서 기반 검증과 서브에이전트 리뷰가 실제 버그/드리프트를 얼마나 줄이는지 사례 더 찾아보기
- Visual Companion이 실무 PM/디자이너 협업에서 어느 정도까지 유용한지 직접 확인해 보기

## 작업 메모
이번 글은 단순 툴 소개 글로 읽기엔 아까웠습니다.

오히려 저는 이 글을 읽고 나서,
요즘 AI 코딩의 진짜 문제는 “모델이 더 똑똑하냐”보다 **일의 순서를 어떻게 묶어 두느냐**에 있다는 생각이 더 강해졌습니다.

AI가 바로 코드를 쓰게 두면 빠르기는 합니다.
하지만 빠른 것과 덜 틀리는 것은 다른 문제입니다.

그래서 Superpowers가 보여주는 방향은 꽤 설득력이 있었습니다.

- 먼저 묻고
- 문서로 남기고
- 계획으로 쪼개고
- 따로 검토한다

이 흐름은 어쩌면 새롭다기보다,
사람 팀에서 당연히 하던 일을 이제 **AI에게도 강제하기 시작했다**는 쪽에 더 가깝습니다.

그리고 그게 AI 시대에는 오히려 더 중요해질 것 같습니다.
