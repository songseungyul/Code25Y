## 한 줄 요약
이 영상은 Codex의 `/goal` 모드를 잘 쓰려면 **좋은 프롬프트 한 줄보다 종료 조건이 있는 목표, 빠른 검증 루프, 외부 메모리 파일 구조가 더 중요하다**는 점을 강조합니다. 그래서 이번 글은 내용을 완전히 정리한 기록이라기보다, **직접 실험하면서 더 공부해야 할 질문들**을 남겨두는 메모에 가깝습니다.

## 왜 이 영상을 공부할 목록에 넣었나
이 영상은 단순히 “Codex가 좋다”는 소개가 아니라,
**오래 돌아가는 에이전트를 실제로 굴릴 때 어디서 실패하는지**를 꽤 구체적으로 짚어 줍니다.

특히 지금까지는 AI에게:

- 코드 좀 더 좋게 해줘
- 알아서 개선해줘
- 더 깔끔하게 정리해줘

같이 두루뭉술하게 말해도 어느 정도는 결과가 나왔습니다.
그런데 `/goal`처럼 며칠씩 반복 실행될 수 있는 모드에서는,
그 두루뭉술함이 오히려 **무한 루프**나 **너무 빠른 종료**를 만든다는 점이 핵심이었습니다.

그래서 이 영상은 “바로 다 이해했다”기보다,
앞으로 실제 프로젝트에 적용해 보면서 검증해야 할 주제가 많다고 느껴져서 `todo-study`로 남깁니다.

## 지금 이해한 핵심

### 1. `/goal` 모드는 본질적으로 반복 루프다
영상에서 가장 먼저 박아 두는 전제는 이것입니다.

에이전트는:
1. 행동한다
2. 결과를 채점한다
3. 목표를 만족했는지 확인한다
4. 만족하지 못하면 다시 반복한다

즉 `/goal`은 단발성 요청이 아니라 **반복 평가 구조**입니다.
그래서 목표가 모호하면 루프가 의미를 잃습니다.

영상에서는 실패 패턴을 두 가지로 설명합니다.

- **too early**: 조금 하다가 더 할 게 없다고 보고 너무 빨리 끝남
- **never stop**: 만족 불가능한 목표를 향해 끝없이 수정함

둘 다 결국은 **종료 조건이 없는 목표**에서 나온다는 설명이었습니다.

### 2. 좋은 목표는 측정 가능해야 한다
예시도 분명했습니다.

- 나쁜 목표: `make my code better`
- 더 좋은 목표: 특정 파일 실행 시간을 20% 줄이되 테스트 회귀는 없게 만들기

이 차이는 단순한 표현 차이가 아니라,
에이전트가 **언제 멈춰야 하는지 스스로 판단할 수 있느냐**의 차이입니다.

특히 인상적이었던 부분은,
정성적인 목표도 체크리스트로 바꾸면 사실상 정량적 목표처럼 다룰 수 있다는 설명입니다.

예를 들어 논문 포맷 변경 작업처럼 모호해 보이는 일도,
규칙들을 체크리스트로 쪼개면:

- 총 몇 개 규칙이 있고
- 몇 개를 만족했는지
- 언제 종료할지

를 분명히 만들 수 있습니다.

### 3. 피드백 루프는 짧을수록 좋다
두 번째 원칙은 **검증 루프를 짧게 만드는 것**입니다.

에이전트가 자기 행동의 품질을 높이려면,
변경 뒤에 바로 테스트하거나 채점할 수 있어야 합니다.

영상의 요지는 간단합니다.

- 같은 시간에 1~2번 시도하는 것보다
- 수십 번 시도할 수 있는 환경이

에이전트 품질을 훨씬 더 끌어올린다는 것입니다.

다만 여기서 중요한 건 무조건 빠르게만 만드는 게 아니라,
**점수의 신뢰도를 유지하면서 가속해야 한다**는 점이었습니다.

즉 더 가벼운 평가셋, 더 짧은 테스트, 샘플링된 데이터셋을 써도,
그 결과가 원래 문제를 꽤 잘 대표해야 의미가 있습니다.

### 4. 외부 메모리가 필요하다
세 번째 원칙은 **긴 실행 시간 동안 맥락을 에이전트 머릿속에만 두지 말라**는 것입니다.

영상에서는 이를 마크다운 파일 기반의 외부 메모리로 설명합니다.
권장 예시는 대략 이렇습니다.

- `plan.md`: 상위 계획
- `experiment.md`: 시도한 것과 결과 기록
- `notes.md`: 실시간 사고 메모

여기서 특히 `experiment.md`를 가장 중요하게 말한 점이 눈에 띄었습니다.
왜냐하면 사람과 에이전트가 같이 볼 수 있는,
비교적 신뢰할 만한 실행 기록의 중심축이 되기 때문입니다.

## 아직 제대로 이해하지 못한 것
이제부터가 실제로 더 공부해야 할 부분입니다.

### 1. “측정 가능한 목표”를 내 작업에 어떻게 번역할까
영상의 예시는 성능 20% 개선처럼 비교적 명확합니다.
그런데 실제로 내가 자주 하게 될 작업은 더 애매할 수 있습니다.

예를 들면:
- 문서 구조 개선
- 코드 가독성 개선
- 블로그 자동화 흐름 개선
- 리팩터링 안정성 확보

이런 것은 어떻게 체크 가능한 목표로 바꿔야 할지 더 연구가 필요합니다.

지금 떠오르는 질문:
- 정성적 목표를 체크리스트로 바꾸는 기준은 무엇인가?
- 체크리스트 항목은 누가 만들고 얼마나 세분화해야 하나?
- 너무 잘게 쪼개면 오히려 목표를 잃지 않을까?

### 2. 빠른 피드백 루프를 어떤 단위에서 설계해야 할까
영상은 “짧게 유지하라”고 말하지만,
실무에서는 무엇을 줄여야 하는지가 더 중요합니다.

궁금한 점:
- 단위 테스트 중심으로 돌릴지
- 샘플 데이터셋을 따로 만들지
- 전체 파이프라인 대신 부분 검증을 먼저 할지
- 속도와 신뢰도 사이에서 어디까지 줄일 수 있을지

특히 이 부분은 실제 프로젝트마다 다를 가능성이 커서,
작은 예제로 직접 실험해 봐야 감이 올 것 같습니다.

### 3. 외부 메모리 파일을 어떤 형식으로 유지해야 가장 잘 작동할까
`plan.md`, `experiment.md`, `notes.md` 구조는 이해되지만,
실제로 잘 굴러가게 만들려면 형식 설계가 꽤 중요해 보입니다.

예를 들어 `experiment.md` 안에서:
- 시도한 가설
- 실행한 명령
- 결과 요약
- 다음 액션

을 어떤 형태로 고정해야 읽기와 갱신이 쉬운지 더 봐야 합니다.

또 질문이 남습니다.
- 한 파일이 너무 길어지면 어떻게 분리할까?
- 에이전트가 항상 최신 상태를 잘 읽게 하려면 어떤 헤더/섹션 구조가 좋을까?
- 사람이 중간에 개입할 때 어디를 수정하는 게 가장 안전할까?

### 4. Codex `/goal`과 다른 에이전트 도구들의 차이는 무엇일까
영상은 Codex 중심이지만,
비슷한 아이디어는 Claude Code, Hermes, OpenCode 같은 다른 도구에도 적용될 수 있어 보입니다.

그래서 더 공부해 볼 것:
- 어떤 도구가 긴 루프 실행에 더 강한지
- 외부 메모리와 체크리스트를 어떤 식으로 읽는지
- 목표 종료 판정이 얼마나 안정적인지
- 로그와 중간 상태를 사람이 관찰하기 쉬운지

## 나중에 직접 해보고 싶은 실험
이 영상은 결국 직접 돌려 봐야 이해가 될 유형입니다.
그래서 아래 실험을 해 보고 싶습니다.

### 실험 1. 작은 코드 개선 작업에 `/goal` 스타일 적용
예시:
- 특정 함수 실행 시간 10~20% 줄이기
- 기존 테스트 모두 통과 유지
- 실험 기록을 `experiment.md`에 남기기

보고 싶은 것:
- 목표를 얼마나 구체적으로 써야 멈추는지
- 에이전트가 어떤 방식으로 스스로 채점하는지
- 체크 기준이 흐리면 정말 루프가 망가지는지

### 실험 2. 정성적 작업을 체크리스트로 바꾸기
예시:
- 블로그 포스트 템플릿 개선
- 문서 스타일 통일
- README 품질 개선

보고 싶은 것:
- 체크리스트 변환만으로 종료 조건이 명확해지는지
- 사람이 보기에도 납득되는 품질 기준이 되는지

### 실험 3. 외부 메모리 파일 형식 비교
비교 대상:
- `plan.md`만 쓰는 경우
- `plan.md + experiment.md`
- `plan.md + experiment.md + notes.md`

보고 싶은 것:
- 어떤 구성이 가장 덜 꼬이는지
- 에이전트가 중복 시도를 덜 하는지
- 사람이 중간 개입하기 쉬운지

## 이 영상을 보고 바로 적용해 볼 수 있는 체크리스트 초안
나중에 직접 시도할 때는 아래 질문부터 먼저 체크해 보고 싶습니다.

- 이 목표는 숫자나 완료 조건으로 표현 가능한가?
- 종료 조건이 명확한가?
- 실패/성공을 빠르게 판별할 테스트가 있는가?
- 테스트 시간을 더 줄일 수 있는 대체 루프가 있는가?
- `experiment.md`에 무엇을 남길지 먼저 정했는가?
- 사람이 중간에 읽어도 이해되는 기록 구조인가?

## 작업 메모
이 영상이 좋았던 이유는 `/goal`을 신기한 기능처럼 소개하기보다,
**에이전트를 오래 굴릴 수 있게 만드는 운영 원칙**으로 설명했다는 점입니다.

지금 단계에서 제일 중요한 건 “바로 큰 프로젝트에 쓰기”보다,
작고 실패해도 괜찮은 문제 하나를 골라:

- 목표를 더 명확하게 쓰고
- 짧은 피드백 루프를 만들고
- 외부 메모리를 남기게 한 뒤
- 실제로 어디서 막히는지 관찰하는 것

이라고 느꼈습니다.
